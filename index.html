Era 01
Building the Toolbox
1999 – 2005
One painful problem.
One elegant solution.
🔍
SQL Compare
The founding product
Schema comparison and sync for SQL Server. Built to solve the founders' own pain — comparing schemas was manual, slow, and error-prone.
SQL Server Active
1999
📊
SQL Data Compare
Schema's natural companion
Customer demand for SQL Compare immediately extended to the data inside databases. Compare and sync row-level data between SQL Server instances.
SQL Server Active
2001
⚡
ANTS Perf Profiler
Redgate enters .NET
Line-level performance profiling for .NET apps. Redgate's first expansion beyond SQL Server — same "painful problem, elegant solution" playbook applied to a new audience.
.NET Active
2002
🧠
ANTS Memory Profiler
Hunt memory leaks
Detect memory leaks and optimize allocation in .NET. Paired with ANTS Performance Profiler to form a complete diagnostics suite for .NET developers.
.NET Active
2004
Why We Pivoted
"Individual tools aren't enough — teams need the whole workflow"
Customers were buying SQL Compare, then immediately asking what else existed for their daily database work. The opportunity: build a complete suite, not isolated utilities.
Era 02
The Developer Suite
2006 – 2010
From utilities to an integrated suite.
✍️
SQL Prompt
Intelligent SQL autocomplete
IntelliSense-style code completion, formatting, and snippet management inside SSMS. Developers who tried it once rarely returned to bare SSMS.
SQL Server Active
2006
🔭
.NET Reflector
First major acquisition
Acquired from Lutz Roeder — the de-facto .NET decompiler used by millions. Proved that acquisition was a viable growth lever, not just organic product development.
.NET Acquired
2008
📡
SQL Monitor
Real-time SQL Server health
Real-time performance monitoring and alerting for SQL Server estates. Later rebranded Redgate Monitor when it expanded to PostgreSQL and Azure — the name change itself a signal.
SQL Server → RG Monitor
2009
🌿
SQL Source Control
Databases in version control
Version-control SQL Server schemas directly from SSMS. The seed for everything that followed — it proved that databases could join the DevOps era alongside application code.
Git Active
2010
Why We Pivoted
"Databases got left behind in DevOps — and nobody was fixing it"
Application code had CI/CD pipelines. Databases still had risky, manual, Friday-night deployments. Redgate's decision: become the company that completes the DevOps story for data teams.
Era 03
Database DevOps
2011 – 2018
Bring databases into the pipeline.
🧪
SQL Test
Unit testing for SQL Server
Built on open-source tSQLt. Without automated tests, nobody could confidently automate database deployments — SQL Test completed the DevOps loop for database code.
Testing Active
2011
🚀
SQL Change Automation
ReadyRoll acquired
ReadyRoll acquisition gave Redgate a migrations-based CI/CD approach for databases, integrating directly with Azure DevOps and Jenkins — filling the last gap in automated delivery.
CI/CD Acquired
2015
🗂️
SQL Data Catalog
Classify sensitive data
Classify and document PII and sensitive data across SQL Server estates. A direct response to GDPR and growing enterprise governance demands — the compliance angle Redgate had been building toward.
GDPR Active
2017
Why We Pivoted
"Open source isn't a threat — it's the biggest growth channel we've never used"
Flyway had millions of users across dozens of databases. Acquiring it doubled Redgate's total addressable market overnight and gave access to a developer community that SQL Server-only tools could never reach.

Product Evolution

The Road to
Test Data Manager

Six distinct stages, spanning 15+ years — each one solving the problem the previous product left behind. Here's every decision, why it was made, and which databases each product supported.

6 stages  ·  SQL Backup → SQL Clone → SQL Data Masker → SQL Provision → Redgate Clone → Test Data Manager
Scroll to follow the evolution
Stage 01 · Origin
SQL Backup Pro
c. 2007
The original approach to test data provisioning: take a production backup, restore it to a dev or test environment. Simple, reliable, and slow.
Databases Supported
SQL Server
⚠ Slow full copies · Huge storage footprint · No data masking — can't safely use real production data in test environments
~2007
→
Why we moved on
"Restoring a 200 GB backup to give a dev a sandbox took hours and killed storage. Teams needed instant copies."
Stage 02
SQL Clone
c. 2017
Virtual database cloning using Windows storage virtualisation (copy-on-write). A developer could get a fresh, isolated copy of a database in seconds — not hours — with a fraction of the storage footprint.
Databases Supported
SQL Server
⚠ Still no masking — cloning production data means cloning PII. Windows-only virtualisation had no path to cloud or containers.
~2017
→
Why we moved on
"Fast clones were useless if you couldn't clone production. GDPR in 2018 made masking an urgent requirement, not a nice-to-have."
Stage 03
SQL Data Masker
c. 2014 – 2018
Mask, anonymise, and obfuscate sensitive production data so it could be safely used in non-production environments. The compliance answer to the problem SQL Clone exposed — teams could now clone and mask.
Databases Supported
SQL Server Oracle
⚠ A separate product from SQL Clone — teams had to manually stitch together two tools with no shared workflow. Error-prone and time-consuming.
~2014
→
Why we moved on
"Customers shouldn't have to use two separate products and stitch them together manually. We needed to unify the provisioning journey."
Stage 04
SQL Provision
c. 2018 – 2021
SQL Data Masker + SQL Clone unified into a single provisioning workflow. For the first time, teams could mask production data and instantly provision virtual copies — all from one product.
Databases Supported
SQL Server
⚠ Still SQL Server and Windows only. Cloud adoption and container-based workflows were accelerating — the Windows virtualisation model had a ceiling.
~2018
→
Why we moved on
"Teams were moving to Docker and cloud. A Windows-only virtualisation approach couldn't follow them. We needed to rethink the technology from the ground up."
Stage 05
Redgate Clone
c. 2021 – 2022
A rebuild from scratch — container-based database virtualisation using Docker. Fast, cloud-friendly, and no longer tied to Windows storage APIs. Opened the door to multi-platform database support.
Databases Supported
SQL Server PostgreSQL
⚠ Cloning alone still isn't the full picture. Customers needed masking, provisioning automation, governance, and self-service — not just one piece of the puzzle.
~2021
→
Why we moved on
"Cloning is one capability in a much larger need. Teams wanted a single platform — not a tool that solves provisioning but leaves masking, governance, and self-service to someone else."
Current Platform
Stage 06
Test Data Manager
2022 – Present
The complete test data platform: mask production data, provision virtual copies, manage data across environments, enforce governance — all from one unified product. Every previous stage's lesson built into one.
Databases Supported
SQL Server PostgreSQL Oracle MySQL
2022
