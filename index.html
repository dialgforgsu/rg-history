Era 01
Building the Toolbox
1999 – 2005
One painful problem.
One elegant solution.
🔍
SQL Compare & SQL Data Compare
The founding products
Schema and data comparison for SQL Server. Built to solve the founders' own pain — SQL Compare handled structure (1999), SQL Data Compare extended that to row-level data (2001).
SQL ServerActive
1999 – 2001
⚡
ANTS Perf Profiler
Redgate enters .NET
Line-level performance profiling for .NET apps. Same "painful problem, elegant solution" playbook applied to a new developer audience.
.NETActive
2002
🧠
ANTS Memory Profiler
Hunt memory leaks
Detect memory leaks and optimize allocation in .NET. Paired with ANTS Performance Profiler to form a complete diagnostics suite.
.NETActive
2004
🔒
SmartAssembly
.NET protection
Protect .NET assemblies from reverse engineering with obfuscation, tamper detection, and automated error reporting. Redgate's first .NET security tool.
.NETActive

"Individual tools aren't enough — teams need the whole workflow"
Customers were buying SQL Compare, then immediately asking what else existed. The opportunity: build a complete suite, not isolated utilities.

SQL Clone
Virtual database copies

Product Evolution
The Road to
Test Data Manager
Six stages across 15+ years. Each one solved the problem the previous product left behind.
5 stages SQL Server → Multi-platform
Stage 01
SQL Clone
c. 2017
Virtual database cloning using Windows storage virtualisation (copy-on-write). A developer could get a fresh isolated copy in seconds, not hours, using a fraction of the storage.
Databases Supported
SQL Server
⚠Still no masking — cloning production data means cloning PII. Windows-only, no path to cloud or containers.
~2017
→
Why we moved on
"Fast clones were useless if you couldn't clone production. GDPR in 2018 made masking urgent, not a nice-to-have."
Stage 02
SQL Data Masker
c. 2014 – 2018
Mask, anonymise, and obfuscate sensitive production data for safe use in non-production environments. The compliance answer that made SQL Clone's promise real.
Databases Supported
SQL Server
⚠A separate product from SQL Clone — teams had to manually stitch together two tools with no shared workflow.
~2014
Supporting Tool · 2017
SQL Data Catalog
2017
Before you can mask sensitive data, you need to know where it lives. SQL Data Catalog automated the discovery and classification of PII across SQL Server estates — becoming the essential input to any masking workflow.
Databases Supported
SQL Server
⚠Classification and masking were still separate products — teams had to manually map catalog output into Data Masker rules.
2017
→
Why we moved on
"Customers shouldn't stitch together two separate products. We needed to unify the provisioning journey into one workflow."
Stage 03
SQL Provision
c. 2018 – 2021
SQL Data Masker + SQL Clone unified into a single provisioning workflow. Teams could mask production data and provision virtual copies from one product for the first time.
Databases Supported
SQL Server
⚠Still SQL Server and Windows only. Cloud adoption and container-based workflows were accelerating — the virtualisation model had a ceiling.
~2018
→
Why we moved on
"Teams were moving to Docker and cloud. A Windows-only virtualisation approach couldn't follow them. We needed to rethink from the ground up."
Stage 04
Redgate Clone
c. 2021 – 2022
A rebuild from scratch — container-based database virtualisation using Docker. Fast, cloud-friendly, no longer tied to Windows storage APIs. Opened the door to multi-platform support.
Databases Supported
SQL ServerPostgreSQLMySQLOracle
⚠Installation required Spoon containerisation and Kubernetes clusters — infrastructure most customers didn't have or couldn't operate. Cloning alone also left masking and governance unsolved.
~2021
→
Why we moved on
"The technology was right but the installation was too hard. Most customers couldn't run Kubernetes just to get a clone. And cloning alone still wasn't the full answer."
Current Platform
Stage 05
Test Data Manager
2022 – Present
Mask, subset, and govern test data — safely and at scale. Works alongside SQL Clone, but is a fully separate product focused purely on the data, not the delivery mechanism.
Databases Supported
SQL ServerPostgreSQLOracleMySQL

Database Deployment Evolution
The Path to
Flyway
From schema comparison to the world's most popular open-source database migration tool — how Redgate's deployment story evolved over 20 years.
6 stages 1999 – 2019 30+ databases
Stage 01 · Origin
SQL Compare & SQL Data Compare
1999 – 2001
The foundation of database change management — understanding exactly what is different between environments, both in schema and in data. Before you can deploy a change, you need the complete picture.
Databases Supported
SQL Server
⚠Reactive comparison only — showed the current state of differences but didn't track how they accumulated over time or help automate what to do about them.
1999 – 2001
→
Why we moved on
"Teams could compare both schema and data across environments — but had no way to track how they got that way. They needed version control: history, accountability, rollback."
Stage 02
SQL Source Control
2010
Version-control SQL Server schemas directly from SSMS. Database changes could now be tracked, branched, and reviewed the same way application code was — for the first time.
Databases Supported
SQL Server
⚠Captured history but didn't automate deployments — teams still manually applied changes in each environment down the pipeline.
2010
→
Why we moved on
"Source control gave you history. But teams needed the pipeline to automatically deploy those changes from dev through to production."
Stage 03
SQL Change Automation
c. 2013
State-based CI/CD for databases — automatically compare source to target and deploy the difference. Integrated with Azure DevOps, TeamCity, and Jenkins to bring database changes into the same pipelines as application code.
Databases Supported
SQL Server
⚠State-based deployment worked well for DBAs but didn't match how many developers thought — they wanted ordered, versioned migration scripts.
~2013
→
Why we moved on
"Developers wanted migrations — explicit, versioned scripts they controlled. ReadyRoll already had this. We acquired it to bring that approach in-house."
Stage 04 · Acquired
ReadyRoll
c. 2015
Migrations-based database deployments — ordered, versioned scripts that gave developers full control over exactly what ran, in what order, in every environment. Acquired to complement SQL Change Automation's state-based approach.
Databases Supported
SQL Server
⚠A standalone acquisition that needed deeper platform integration and expansion beyond SQL Server to realise its full potential.
~2015
→
Why we moved on
"ReadyRoll needed to become a first-class Redgate product — integrated with the full platform and positioned as Redgate's unified deployment answer."
Stage 05
Redgate Deploy
c. 2018
ReadyRoll rebranded and evolved into Redgate Deploy — Redgate's unified database deployment product, combining migrations-based and state-based approaches with deep CI/CD pipeline integrations.
Databases Supported
SQL ServerAzure SQL
⚠Still primarily SQL Server-focused. The open-source community had already chosen Flyway as the standard for cross-platform, developer-first migrations.
~2018
→
Why we moved on
"Flyway had millions of users across every major database. Acquiring it gave us the platform-agnostic, developer-first migration tool — and the community that came with it."
Current Platform
Stage 06 · Acquired
Flyway
2019 – Present
Acquired from creator Axel Fontaine — the world's most popular open-source database migration tool. Flyway completed Redgate's deployment story: developer-first, open-source, running on every major database platform.
